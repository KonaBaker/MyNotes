### skill

**skill以及CLAUDE.md**

两者是相似的。

- skill是任务级别，告诉AI这类任务怎么做。包括操作步骤、代码模板、以及一些需要注意的易错点等。
- CLAUDE.md是项目级别，是告诉AI项目背景，架构，代码规范等等。

两者都是给AI的上下文进行注入的。

**MCP是一种能力工具，是一种扩展**

MCP是AI可以进行调用的工具。

#### Q1. 各种编程工具，上下文满了会发生什么？

**现象层面：** 不同工具表现不同——Claude Code 会自动触发 compaction（把前面的对话压缩成摘要），Cursor 通常做截断或滑窗，有些工具直接报错退出。但症状是共性的：模型开始"失忆"，重复问之前已经回答过的问题，代码风格漂移，幻觉率上升，甚至忘记你最初定的需求。

**应对策略：**

- 主动管理而不是被动等爆——用 `/compact`、`/clear` 手动收口
- 把关键约束沉淀到 `CLAUDE.md`、`.cursorrules`，避免依赖对话记忆
- 长任务拆分阶段，每个阶段结束前让 agent 自己总结输出"交接文档"
- 用 subagent 隔离上下文：主 agent 调度，子 agent 处理具体细节

**面试加分项：** 提到"上下文窗口大不等于有效注意力大"——长上下文里中间部分（lost in the middle）的召回率会下降，所以即使没满也要主动管理。

#### Q2. 额度受限的情况下，贵/便宜/文本/多模态 agent 怎么用？

核心原则是 **"按任务匹配能力，按能力分配预算"**：

| 任务类型                                             | 推荐                    |
| ---------------------------------------------------- | ----------------------- |
| 脚手架、批量改名、格式化、boilerplate/实现代码的主力 | 便宜模型/中端模型       |
| 架构设计、复杂重构、调试难 bug                       | 贵的大模型（Opus 级别） |
| renderdoc分析性能，分析某一张纹理                    | 多模态                  |
| 分析日志或者codereview                               | 文本足够                |

**实战流水线：** 先让便宜模型出草稿/列方案，贵模型只做关键决策和最终 review。游戏开发中，UI 还原、美术 review、关卡截图分析这类场景多模态是刚需，其他能省就省。

**面试加分项：** 提到 **prompt caching**——同一个 codebase 反复对话时，缓存命中能把成本压到 1/10。

#### Q3. 如果 agent 在一件事上反复犯错怎么办？

**最重要的一条：停下来，不要继续重试。** 反复 retry 只会让上下文污染更严重。

诊断顺序：

1. **看它实际做了什么**，不是看它说自己做了什么——读 diff、跑测试
2. **判断错误类型**：
   - 缺上下文？→ 补具体代码片段、版本信息、类似实现的例子
   - prompt 太模糊？→ 给输入输出示例，明确边界条件
   - 模型能力不够？→ 换更强的模型，或者人工干预关键步骤
   - 训练数据过期？→ 比如 UE5.4 的新 API，要直接贴文档
3. **清理上下文重开**：把任务隔离到新会话，只带必要信息
4. **最后兜底**：自己写一遍正确版本，让 agent 学习这个 pattern 后再做下一个类似任务

**面试加分项：** 把反复犯错的 case 沉淀到 `CLAUDE.md` 或 rules 文件里，这是团队级别的资产。

#### Q4. UE 代码很长，上下文不够怎么办？

UE 源码动辄百万行，硬塞肯定不行。策略：

**不要 load 整个文件，要按需检索：**

- 用 ripgrep / 语义搜索定位符号，比如找一个 `UCharacterMovementComponent::PerformMovement` 的调用链
- Agent 工具应该支持 `str_replace` 这类不需要 load 全文就能改的操作
- 文件 view 用 `view_range` 只看相关行

**架构信息预先编码：**

- 在 `CLAUDE.md` 里写清楚关键模块的入口、继承链、宏（`UCLASS`/`UFUNCTION`/`GENERATED_BODY`）的处理约定
- 列出常踩的坑：反射系统、GC、热重载限制——这些 agent 默认不知道

**任务分而治之：**

- 一次只 focus 一个 system（如只动 Animation 不动 Networking）
- 用 subagent 并行探索：一个查调用方，一个查实现，主 agent 汇总

**面试加分项：** 提到 UE 的 module 化结构天然适合分块——按 `.Build.cs` 边界切，避免跨模块上下文。

#### Q5. 写过 skill 么，skill 太长怎么办？

**Skill 是什么先说清楚：** 它是给 AI agent 的"任务手册"，把某类工作的最佳实践、踩坑记录、代码模板封装成可复用的 `SKILL.md`，agent 遇到匹配任务时自动加载执行。

**Skill 太长的本质：** 一次性塞进上下文不划算，要用 **渐进式披露（Progressive Disclosure）**：

```
skill-name/
├── SKILL.md          ← 始终加载，控制在 500 行内
├── references/       ← 按需加载的详细文档
│   ├── platform-a.md
│   └── platform-b.md
├── scripts/          ← 可执行脚本，不占上下文
└── assets/           ← 模板、字体等
```

**三级加载：**

1. **元数据**（name + description）始终在上下文里，~100 词
2. **SKILL.md 主体** 触发时才加载
3. **bundled resources** 用到才看

**实操原则：**

- SKILL.md 只放"决策树和工作流"，把"详细规格"丢到 references
- 多平台/多场景时，按变体拆分（如 `aws.md`、`gcp.md`），agent 只读相关那个
- 大于 300 行的 reference 文件加 TOC，方便定位

**reference名字相近怎么办？**

- 子目录隔离。
- 命名的时候加上模块的前缀。
- 在skill头部写明在什么情况下去读某一个具体路径下的reference。

**面试加分项：** 说出"Skill 设计像写代码——单一职责、关注点分离，主文件是接口，references 是实现"。
